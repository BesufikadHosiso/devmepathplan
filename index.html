The mindset that changes everything

Juniors ask "what do I build next?" Seniors ask "what foundation do I need so everything I build next is easy?" This document gives you that foundation — for DevPath specifically.

◆
The Tesla rule: You're not building something new. You're building something familiar but dramatically better. Habit trackers exist. Focus timers exist. Learning logs exist. But nobody combined all three specifically for developers who keep restarting. That's your moat. Never forget it when you feel like quitting.
◆
Build for yourself first: You ARE the user. When you hit a problem using DevPath, fix it immediately. Your own frustration is the best product feedback you'll ever get. Ship fast, use daily, improve daily.
◆
The order is the secret: Most juniors fail not because they can't code — but because they code in the wrong order. They build the roof before the foundation. This document gives you the exact order. Follow it, don't skip steps.
Architecture

The full folder structure

Create this before anything else. This is your map. Every file you create for the rest of the project has a home here. Changing structure later is expensive — get it right now.

devpath/ — project root
devpath/ ├── src/ ├── components/ ├── ui/ ← dumb, reusable atoms │ ├── Button.jsx │ ├── Card.jsx │ ├── Badge.jsx │ ├── Input.jsx │ └── Modal.jsx ├── layout/ ← structure every page shares │ ├── Layout.jsx │ ├── Sidebar.jsx │ └── Navbar.jsx └── features/ ← smart feature components ├── StreakBar.jsx ├── MoodPicker.jsx ├── TaskList.jsx ├── FocusTimer.jsx └── LogEntry.jsx ├── pages/ ← full screens, assembled from components ├── Dashboard.jsx ├── Focus.jsx ├── Log.jsx ├── Stats.jsx └── Path.jsx ├── context/ ← app brain / global state └── AppContext.jsx ├── hooks/ ← reusable logic ├── useApp.js ├── useStreak.js └── useTimer.js ├── utils/ ← pure helper functions ├── dates.js └── storage.js ├── App.jsx ← router lives here └── main.jsx ← entry point, wraps in Provider ├── tailwind.config.js ← design tokens ├── index.html └── package.json
◆
The 3-layer rule: ui/ = no logic, just looks. features/ = logic + no page layout. pages/ = assembles features into screens. Never skip layers. A Button has no useEffect. A Dashboard has no raw divs.

Step 02

Design tokens — one source of truth

Before writing any component, define your brand in tailwind.config.js. This means every color, every font, every spacing decision lives in one file. If DevPath's brand ever changes — you change it here and the whole app updates.

tailwind.config.js
export default { content: ["./index.html", "./src/**/*.{js,jsx}"], theme: { extend: { colors: { // DevPath brand colors — change here = changes everywhere night: '#0a0c0f', surface: '#0f1117', border: 'rgba(255,255,255,0.07)', streak: '#6EE7B7', // green — reward, progress focus: '#A78BFA', // purple — concentration mood: '#FCD34D', // amber — energy, mood muted: '#8b90a0', // secondary text ghost: '#4a4f5e', // tertiary text }, fontFamily: { display: ['Syne', 'sans-serif'], body: ['DM Sans', 'sans-serif'], mono: ['JetBrains Mono', 'monospace'], }, }, }, plugins: [], }
◆
Rule: Never write text-[#6EE7B7] in your components. Always write text-streak. The named token carries meaning. The hex value carries nothing.
Step 03

UI primitives — your Lego bricks

Build 4 dumb components first. These have zero logic — just props and Tailwind classes. Every page you build later uses these. Build them once, use them everywhere.

components/ui/Button.jsx
// This component does ONE thing: look like a button. // No fetch. No state. No useEffect. Ever. const variants = { primary: 'bg-streak text-night font-semibold hover:bg-streak/90', ghost: 'border border-white/10 text-muted hover:bg-white/5', danger: 'bg-red-500/10 border border-red-500/20 text-red-400', } export default function Button({ children, variant = 'ghost', onClick, disabled, className = '' }) { return ( <button onClick={onClick} disabled={disabled} className={` px-4 py-2 rounded-lg text-sm transition-all duration-150 disabled:opacity-40 disabled:cursor-not-allowed ${variants[variant]} ${className} `} > {children} </button> ) }
◆
The dumb component rule: If you find yourself writing useEffect or fetch inside Button, Card, Badge, or Input — stop. Extract that logic into a feature component or a hook. Dumb components are reusable forever. Smart ones are reusable never.

Build these 4 before anything else:

File Props What it does

Step 04

Layout shell — build once, use forever

This is the container every page lives inside. Sidebar + top nav + a content area. You build this once and never touch it again. Every page just slots in the middle.

components/layout/Layout.jsx
import Sidebar from './Sidebar' export default function Layout({ children }) { return ( <div className="flex min-h-screen bg-night"> <Sidebar /> <main className="flex-1 min-w-0 overflow-y-auto"> {children} </main> </div> ) } // Usage in App.jsx: // <Layout><Dashboard /></Layout> // <Layout><Focus /></Layout> // The sidebar never re-renders. Only the middle swaps.
● Phase 2 — Brain
Step 05 — The most important step

AppContext — your app's brain

This is the single most important file in DevPath. All your data lives here. Tasks, streak, mood, log entries. Every page reads from here. localStorage saves it automatically — no backend needed.

context/AppContext.jsx
import { createContext, useContext, useReducer, useEffect } from 'react' // 1. The shape of ALL your app's data const initialState = { tasks: [], // today's learning tasks streak: 0, // current day streak mood: null, // 'rough' | 'okay' | 'good' | 'pumped' log: [], // [{date, text, tag}] sessions: [], // [{date, duration, mood}] path: [], // [{id, title, done, order}] } // 2. Reducer — every action that can change state function reducer(state, action) { switch (action.type) { case 'ADD_TASK': return { ...state, tasks: [...state.tasks, action.payload] } case 'TOGGLE_TASK': return { ...state, tasks: state.tasks.map(t => t.id === action.payload ? { ...t, done: !t.done } : t ) } case 'SET_MOOD': return { ...state, mood: action.payload } case 'ADD_LOG': return { ...state, log: [action.payload, ...state.log] } case 'INCREMENT_STREAK': return { ...state, streak: state.streak + 1 } case 'LOAD': return action.payload default: return state } } // 3. Create context const AppContext = createContext(null) // 4. Provider — wraps the whole app export function AppProvider({ children }) { const [state, dispatch] = useReducer(reducer, initialState) // Load from localStorage on first open useEffect(() => { const saved = localStorage.getItem('devpath') if (saved) dispatch({ type: 'LOAD', payload: JSON.parse(saved) }) }, []) // Save to localStorage on every state change useEffect(() => { localStorage.setItem('devpath', JSON.stringify(state)) }, [state]) return ( <AppContext.Provider value={{ state, dispatch }}> {children} </AppContext.Provider> ) } // 5. Custom hook — how every component reads the brain export function useApp() { return useContext(AppContext) } // Usage in any component: // const { state, dispatch } = useApp() // dispatch({ type: 'ADD_TASK', payload: { id: Date.now(), text: 'Build form', done: false } })
◆
Why useReducer instead of useState? When your state has multiple related pieces that change together, useReducer keeps all the update logic in one place. You can see every possible state change at a glance. That's architecture — not magic.
◆
Wrap in main.jsx: Put <AppProvider> around your <App /> in main.jsx. This means every single component in your app can call useApp() and get the data — no prop drilling ever.

Build the pages — in this exact order

Now the fun part. By this point you have: tokens, atoms, layout, and a brain. Every page is just assembly. Build them in this order — each one teaches you something you need for the next.

◆
The page rule: A page component should read like a newspaper — you can skim it and understand what it shows. If your Dashboard.jsx is 300+ lines of raw HTML, you skipped the component extraction step. Each visual block should be a named component.

Step 07

Routing — connect everything last

Add React Router after all pages exist. Routing is plumbing — build the rooms first, then connect the pipes. This is the last step, and it takes 20 minutes.

App.jsx
import { BrowserRouter, Routes, Route, Navigate } from 'react-router-dom' import Layout from './components/layout/Layout' import Dashboard from './pages/Dashboard' import Focus from './pages/Focus' import Log from './pages/Log' import Stats from './pages/Stats' import Path from './pages/Path' export default function App() { return ( <BrowserRouter> <Layout> <Routes> <Route path="/" element={<Dashboard />} /> <Route path="/focus" element={<Focus />} /> <Route path="/log" element={<Log />} /> <Route path="/stats" element={<Stats />} /> <Route path="/path" element={<Path />} /> <Route path="*" element={<Navigate to="/" />} /> </Routes> </Layout> </BrowserRouter> ) }
⚡ The secrets
What seniors know but rarely share

The things nobody tells juniors

These are not tips. These are the things that separate code that works from code that scales, impresses, and survives a real product lifecycle.

①
Build the empty state first — before the happy path
What does the app look like when there are zero tasks? Zero log entries? Day 1 with no streak? Seniors build this first because it forces you to understand the data shape. Juniors build the happy path and then panic when the data is empty and everything breaks.
②
Functional updates prevent bugs you won't understand
Always write setTasks(prev => [...prev, newTask]) — not setTasks([...tasks, newTask]). The second one captures a stale snapshot of tasks. Inside event handlers and async functions this causes subtle bugs that are nearly impossible to debug. The functional form always reads the latest state. Always.
③
localStorage IS your backend for v1 — use it fully
You don't need Supabase, Firebase, or any backend to ship DevPath v1. localStorage persists data between sessions, works offline, and requires zero setup. The AppContext pattern above handles it for you automatically. Ship the product first. Add a real backend when you have real users who need it.
④
Name things by what they ARE, not what they DO
Not handleData, fetchStuff, processItem, doThing.
Yes: addTask, toggleStreak, logInsight, setMood.

Code that reads like English is professional code. A senior reading your file should understand what it does without running it. This one habit will make you look 2 years more experienced immediately.
⑤
One component, one job — the "and" test
If you describe a component with the word "and" — "it shows the streak AND lets you pick mood" — split it into two components. StreakBar and MoodPicker. This is the single rule that separates spaghetti code from professional architecture. Components with one job are testable, reusable, and understandable in 10 seconds.
⑥
Colocate logic — don't centralise everything
New developers think "I'll put all my helper functions in utils.js" and end up with a 600-line mess. Instead: if the logic belongs to one component, put it in that component or a hook right next to it. useTimer.js lives next to FocusTimer.jsx. Only genuinely shared utilities go in utils/. Ask yourself: "will another component ever need this?" If no — keep it local.
⑦
Derived state — never store what you can compute
If you have a list of tasks and you want to show "completed count" — don't create a completedCount state. Compute it: const completedCount = tasks.filter(t => t.done).length. Storing derived data creates sync bugs — two pieces of state that should match but drift apart. Your Stats page is entirely derived from tasks, sessions, and log. No new state needed.
⑧
useEffect cleanup — the memory leak you're writing right now
The Focus timer uses setInterval. If you don't return a cleanup function from useEffect, that interval keeps running even after the user navigates away — draining battery, causing bugs, and leaking memory. Always return cleanup for timers, event listeners, and subscriptions.
Avoid these

The 6 mistakes that kill junior projects

MistakeWhat it looks likeThe fix
God component App.jsx is 800 lines with everything inside it Split. One component, one job. Always.
Prop drilling Passing data through 5 components to reach 1 Put it in AppContext. Access it directly anywhere.
Index as key tasks.map((t, i) => <Task key={i} />) Use stable IDs: key={task.id}
Direct mutation tasks.push(newTask) then setState Always return new arrays. Never mutate.
No empty states App crashes or shows nothing when data is empty Handle empty explicitly — always.
No loading states UI freezes while async operations run Show a spinner or skeleton while waiting.

Daily ritual

Your daily build checklist

Paste this at the top of every coding session. It takes 30 seconds and prevents 3 hours of confusion.

before you code — answer these
1. What ONE thing am I building today? (not "work on DevPath") 2. Where does the file I'm creating live in the folder structure? 3. Is this a ui/ atom, features/ component, or pages/ screen? 4. Does the data I need live in AppContext already? 5. What is the empty state for this component? ── while coding ── 6. Am I about to write the word "and" in my component description? If yes → split into two components first. 7. Am I passing props more than 2 levels deep? If yes → put it in context. 8. Am I storing something I could compute instead? If yes → delete the state, derive it. ── after you code ── 9. Log one sentence: what did I actually learn or build today? (Open DevPath. Use it. You built it for this.)
